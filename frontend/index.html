Vector DB Architecture

Compare the leading vector indexes and database services side-by-side to understand when to deploy each:

FAISS (Facebook AI Similarity Search)
Type: In-Memory Library
Pricing: 100% Free / Open Source
  • Pros: Extremely fast, low overhead, no network latency, offline execution, supports GPU acceleration.
  • Cons: No built-in storage persistence (requires manual file serialization), no horizontal scalability, must rebuild indexes entirely on updates.
  • Best For: Edge devices, rapid local prototyping, small-to-medium document stores (< 1M records).
Pinecone
Type: Managed Cloud SaaS
Pricing: Free Tier / Serverless Pay-go
  • Pros: Fully managed (zero devops), instant horizontal scaling, real-time index updates, rich metadata filtering.
  • Cons: Closed-source, cloud vendor lock-in, recurring subscription costs, requires persistent internet connection.
  • Best For: Production SaaS apps needing high availability, scale, and fast integration.
Vertex AI Vector Search
Type: Enterprise Cloud Infrastructure
Pricing: Paid GCP Resource
  • Pros: Scans billions of vectors in milliseconds, deep integration with GCP (BigQuery, IAM), enterprise-grade security.
  • Cons: Highly complex setup, high entry cost, overkill for smaller projects.
  • Best For: Multi-million/billion-scale vector databases integrated into broader enterprise GCP ecosystems.

Chunking Strategies

Fixed-Size Character Chunking: Splits text strictly every N characters. While simple, it splits sentences or paragraphs in half, fragmenting context.

Recursive Character Chunking: Iteratively splits text by a sequence of characters (e.g. \n\n, \n, ., ) until chunks are small enough. This preserves paragraph and sentence structures, ensuring semantic coherence.

Sparse vs. Dense Retrieval

Sparse (BM25): Traditional keyword matching. Extremely robust for matching exact terms (IDs, names, error codes, specific keywords), but fails on synonyms or semantic meaning.

Dense (Embeddings): Vector search. Encodes semantic concepts into high-dimensional space. Excels at retrieving conceptually similar documents (e.g., matching "dog" with "canine"), but fails on spelling errors or exact codes.

Hybrid & RRF (Reciprocal Rank Fusion)

Combining sparse and dense search yields the best retrieval performance. We use RRF to merge rankings from FAISS and BM25:

RRF = 1 / (60 + DenseRank) + 1 / (60 + SparseRank)

RRF awards the highest score to items appearing near the top of either list, ensuring both keyword and semantic matches are retrieved.
